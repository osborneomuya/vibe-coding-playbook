# 前端 Vibe 工作流

> 让 AI 产出的前端代码**风格统一、交互完整、状态可控**。

---

## 1) 核心原则

**A：先有数据契约再写页面**
- 接口返回什么结构？哪些字段可能为空？
- 类型定义是否先于 UI 代码存在？
- **没有稳定的数据契约就写页面 = 在沙子上盖楼**

**B：别让用户看到白屏**
- 每个涉及数据的页面都要处理：**loading / error / empty / 正常**
- 接口字段为空或异常时要有兜底渲染，不能直接崩掉

**C：样式必须统一**
- 用项目统一的组件库和 design token，不自己造轮子
- 颜色、间距、字号用变量，不硬编码

---

## 2) 组件结构

AI 最爱把所有逻辑塞进一个文件。你要守住的底线：

- 展示和逻辑**是否分开**？（组件能不能不依赖接口独立渲染）
- 数据获取**是否集中**？（别在子组件深处各自偷偷调接口）
- 单个文件**是否过大**？（超过 300 行就该拆）

目标结构：
- **展示组件**：只接收 props，只负责渲染
- **逻辑 Hook**：数据获取、状态管理、副作用
- **页面组件**：组装展示组件和 hook，处理路由和权限

---

## 3) 状态管理

AI 特别容易制造"状态灾难"——同一份数据在三个地方各存一份。

遇到状态问题，先分类：
- **服务端数据** → 用数据缓存库管理（如 React Query / SWR），不手动存 state
- **全局 UI 状态**（主题/语言/侧边栏）→ 全局 store
- **局部 UI 状态**（弹窗/hover/输入）→ 组件内 state，不上提到全局
- **能从 URL 表达的**（筛选/分页/tab）→ 放 URL params，别放 state

**一句话：服务端数据用缓存库，UI 状态就近管理，URL 能表达的别用 state。**

---

## 4) Prompt 怎么写

别说"写个用户列表页"，而是给 AI 这些输入：

1. **设计参考**：截图 / 设计稿 / 相似页面
2. **组件库**：用什么 UI 库（如 shadcn/ui、Ant Design）
3. **数据契约**：接口返回结构、字段类型、哪些可能为空
4. **交互规格**：操作流程、状态流转、跳转逻辑
5. **边界场景**：loading / error / empty 分别什么样

简单展示页面给 1-2 就够，复杂交互页面建议尽量完整。

---

## 5) 审计 Prompt

```
按以下清单检查本次前端改动：
1. 类型：是否有完整的类型定义？有无 any？
2. 样式一致性：是否使用项目组件库 / design token？有无硬编码样式值？
3. 五态完备：loading / error / empty / 正常 是否都处理了？
4. 数据健壮性：接口字段为空时是否兜底？
5. 组件结构：展示与逻辑是否分离？单文件是否过大？
6. 状态管理：有无重复存储同一份数据？状态归属是否合理？
输出表格：检查项 | 通过/不通过 | 说明。
不通过项直接给修复建议。
```
