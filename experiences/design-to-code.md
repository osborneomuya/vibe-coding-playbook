# 经验分享：AI 百分之百还原设计稿

> 核心思路：**不让 AI 凭空写 UI，而是让设计工具先出静态代码，再让 AI 将"死代码"转化为结构良好、自适应的生产级前端代码。**

---

## 为什么不直接让 AI 看截图写代码？

- AI 从截图还原 UI 看似简单，但**还原度极不稳定**——间距、字号、颜色、圆角等细节经常偏差
- 设计稿转代码工具（Figma Dev Mode / 蓝湖 / MasterGo）能**精确提取设计 Token**（颜色值、间距、字号），比 AI 看图猜值靠谱得多
- 让 AI 做它擅长的事（代码重构、逻辑编排），让工具做它擅长的事（像素级数值提取）

---

## 适用的设计工具

**任何支持"设计稿转代码"的工具都适用本流程**——无论是内置功能还是第三方插件。以下列举几个常见选项：

| 工具 | 转码方式 | 适用场景 |
|------|----------|----------|
| **Figma** | 内置 Dev Mode 导出 + 插件（如 Locofy、Anima） | 团队协作、国际化项目 |
| **蓝湖** | 内置标注 + 代码导出（HTML/CSS） | 国内团队常用，与 Sketch/PS 联动 |
| **MasterGo** | 内置设计转代码 + 插件生态 | 国内替代 Figma，转码能力较新 |
| **Sketch** | 通过插件（如 Sketch2Code）导出 | macOS 设计师常用 |
| **其他** | Zeplin、Locofy、Anima 等独立插件/平台均可 | 只要能导出 HTML/CSS 或框架代码即可 |

> 💡 很多设计工具本身不内置转码功能，但可以通过**安装第三方插件**实现（如 Figma 的 Locofy 插件、Anima 插件等）。选择工具时，优先看其插件生态是否支持你的技术栈。

> 不同工具的转码产物大同小异：**绝对定位 + 硬编码像素值 + 扁平 DOM 结构**。这些"死代码"不能直接用于生产，但**保留了精确的样式数值**，是 AI 改造的完美输入。关键不在于用哪个工具，而在于后续的 AI 转化流程。

## 给设计师的建议：制图方式决定转码质量

**设计师怎么画，直接决定转码工具能导出多干净的代码。** 同样的视觉效果，不同的实现方式会导致转码结果天差地别。

### 核心原则：用工具的原生属性，而非拼凑

| 视觉效果 | ❌ 拼凑式（转码后代码混乱） | ✅ 原生式（转码后代码干净） |
|----------|----------------------------|----------------------------|
| 圆角按钮 | 矩形 + 两个半圆拼接 → 导出 3 个 div | 矩形设置 border-radius → 导出 1 个 div + border-radius |
| 阴影 | 用半透明图层模拟阴影 → 导出多余图层 | 用工具内置 Drop Shadow → 导出 box-shadow |
| 渐变背景 | 多个纯色图层叠加 → 导出多个 div | 用渐变填充 → 导出 linear-gradient |
| 间距控制 | 用空白矩形撑开间距 → 导出多余空 div | 用 Auto Layout / 间距属性 → 导出 gap / padding |
| 图标 | 用基础图形拼凑图标 → 导出一堆 path | 合并为单个矢量图形 → 导出一个完整 SVG |

### 合理的图层结构

- **使用 Auto Layout（自动布局）**：这是转码质量的最大杠杆。Auto Layout 的结构会直接映射为 Flexbox，远比手动定位的图层导出效果好
- **合理分组**：视觉上属于同一组件的元素放在同一个 Group / Frame 中，转码工具才能识别组件边界
- **避免不必要的嵌套**：图层层级越深，导出的 DOM 层级越深，后续 AI 清理的工作量越大

> 💡 **设计师懂一点前端，转码效果会好很多。** 不需要会写代码，但理解"CSS 能原生实现哪些效果"（圆角、阴影、渐变、Flex 布局），就能在设计时自然选择工具的正确实现方式，让转码产物从源头就是干净的。

## 完整流程 SOP

### 第一步：设计稿转代码（工具侧）

1. 在设计工具中选择目标页面/组件
2. 使用内置的"转代码"或"开发模式"功能导出
3. 下载导出的代码文件（通常是 HTML + CSS，部分工具支持 React/Vue）
4. 将下载的代码文件**放入项目的临时目录**（如 `src/_design-draft/`）

**注意事项**：
- 如果设计稿有组件化（Component），优先按组件粒度分别导出，而非导出整个页面
- 导出时选择与项目技术栈匹配的格式（React / Vue / 纯 HTML）

### 第二步：AI 转化为生产级代码

将导出的静态代码喂给 AI，使用以下 Prompt 模板：

```
【任务】将设计稿导出的静态代码转化为生产级前端代码。

【输入文件】
- [粘贴或 @引用导出的代码文件路径]

【硬性要求】
1. **样式保持 100% 不变**：所有颜色值、字号、间距、圆角、阴影必须与原始代码完全一致，禁止"优化"或"调整"任何视觉属性
2. **消除绝对定位**：将 position: absolute 布局改为 Flexbox / Grid，保持视觉效果不变
3. **响应式适配**：固定像素值改为相对单位（rem / %），关键断点添加媒体查询
4. **语义化 HTML**：div 堆叠改为语义标签（header / nav / main / section / article）
5. **使用项目组件库**：[组件库名称，如 Ant Design / Element Plus]，能用现有组件替代的必须替代
6. **使用项目 Design Token**：颜色、间距、字号优先使用项目变量，不硬编码
7. **图片资源**：将内联 base64 或临时路径替换为项目资源引用规范

【禁止事项】
- 禁止修改任何视觉样式值（颜色 / 字号 / 间距 / 圆角）
- 禁止删减设计稿中的任何 UI 元素
- 禁止自行添加设计稿中不存在的元素

【输出要求】
- 输出完整可运行的组件代码
- 标注哪些地方使用了项目组件库替代
- 如果有无法自动转换的部分（如复杂动画、自定义图标），明确列出待人工处理项
```

### 第三步：逐组件对比验收

1. **像素对比**：将 AI 产出的页面截图与设计稿叠加对比（可用浏览器插件如 PerfectPixel）
2. **响应式检查**：在至少 3 个断点（移动端 375px / 平板 768px / 桌面 1440px）下检查布局
3. **交互检查**：hover、active、focus 等状态样式是否完整
4. **组件库一致性**：按钮、输入框、弹窗等是否使用了项目统一组件

## 常见陷阱与应对

### 陷阱 1：AI 擅自"优化"样式

**现象**：AI 觉得某个间距"不协调"，自行调整了 padding 或 margin。

**应对**：
- Prompt 中强调"禁止修改任何视觉样式值"
- 对比时重点检查间距和字号
- 如果 AI 反复改值，在 Prompt 中明确写出"原始值 padding: 12px 16px，必须保持"

### 陷阱 2：绝对定位转 Flex 后布局偏移

**现象**：消除绝对定位后，元素间距或对齐方式发生变化。

**应对**：
- 一次只转一个组件，转完立即截图对比
- 如果偏移较大，让 AI 分步骤转换：先把父容器改成 Flex，再逐个调整子元素

### 陷阱 3：导出代码中的图层命名混乱

**现象**：设计师图层命名为 `Frame 234`、`Group 12`，导出的代码类名无语义。

**应对**：
- **最佳方案**：在导出前与设计师沟通，规范图层命名
- **次选方案**：导出后让 AI 根据视觉内容推断语义，重命名类名。Prompt 补充："根据元素的视觉内容和用途，将无语义的类名（如 frame-234）替换为语义化命名（如 user-card-header）"

### 陷阱 4：字体和图标缺失

**现象**：设计稿使用了特殊字体或自定义图标，导出代码中引用了本地路径。

**应对**：
- 字体：确认项目是否已引入该字体；未引入则与设计师确认替代方案
- 图标：将设计稿中的图标导出为 SVG，放入项目图标体系（如 iconfont / SVG sprite）

### 陷阱 5：响应式断点与项目不一致

**现象**：AI 按常见断点（768px / 1024px）生成了媒体查询，但项目有自己的断点体系。

**应对**：
- Prompt 中明确给出项目断点：如"项目断点：sm=640px, md=768px, lg=1024px, xl=1280px"
- 如果使用 TailwindCSS 等框架，直接指定使用框架内置的响应式类名

## 进阶技巧

### 分层转化策略

对于复杂页面，不要一次性扔给 AI 整页代码，而是**按组件粒度分批转化**：

1. 先拆分页面为独立组件（Header / Sidebar / ContentCard / Footer）
2. 每个组件单独转化 + 验收
3. 最后在页面层组装，处理布局和间距

**好处**：每次 AI 的上下文更聚焦，出错后影响范围小，便于逐个像素对比。

### 建立"设计稿-代码"对照表

第一次转化时，记录设计稿中的颜色 / 间距 / 字号与项目 Design Token 的对应关系：

```
设计稿 #1677FF → var(--color-primary)
设计稿 12px 间距 → var(--spacing-sm)
设计稿 14px 字号 → var(--font-size-base)
```

后续页面转化时，直接将对照表喂给 AI，大幅减少手动修正。

### 利用浏览器 DevTools 快速验收

1. 打开 AI 生成的页面，按 F12 进入 DevTools
2. 使用"元素选择器"逐个点击元素，对比 Computed 面板中的实际样式值与设计稿标注值
3. 重点检查：`font-size`、`line-height`、`padding`、`margin`、`border-radius`、`color`、`background-color`

## 流程总结

```
设计稿（Figma / 蓝湖 / MasterGo）
    │
    ▼ 设计工具转码
静态代码（绝对定位 + 硬编码像素 + 扁平 DOM）
    │
    ▼ AI 转化（使用上方 Prompt 模板）
生产级代码（Flex/Grid + 响应式 + 语义化 + 项目组件库）
    │
    ▼ 像素对比验收
✅ 上线
```

> **核心原则**：工具负责精确提取样式数值，AI 负责代码结构重构，人工负责像素级验收。三者各司其职，才能实现百分之百还原。
